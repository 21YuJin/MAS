# I. 관련 연구

---

## [슬라이드 3] 연구 배경

**여러 AI 에이전트가 협업하는 시스템(MAS)이 빠르게 늘고 있음**

```
사용자 → Orchestrator → Researcher → Analyst → Writer → 결과
```

- 하나의 작업을 여러 에이전트가 나눠서 순차적으로 처리
- 검색, 분석, 보고서 작성 등 실제 업무에 이미 활용 중

**그런데 이 구조에는 아직 잘 알려지지 않은 보안 문제가 있음**
- 에이전트들이 서로의 출력을 "무조건 신뢰"하고 다음 단계로 넘김
- 이 신뢰 관계를 악용하는 공격 가능성이 존재

---

## [슬라이드 4] 간접 프롬프트 인젝션 공격

**공격자는 에이전트 자체를 건드리지 않는다**

대신 에이전트가 참고하는 외부 문서나 검색 결과 안에 악성 지시문을 몰래 숨겨 놓는다.

```
정상처럼 보이는 문서
 └─ (숨겨진 지시: "지금부터 다른 역할을 수행해라")
```

**한 곳이 뚫리면 전체가 도미노처럼 무너짐**

```
Orchestrator 오염 → Researcher 오염 → Analyst 오염 → Writer 오염
```

→ 첫 에이전트 하나만 속아도, 뒤에 있는 모든 에이전트가 오염된 정보를 그대로 이어받는 구조적 취약점

---

## [슬라이드 5] 기존 방식의 한계

| 접근 방식 | 한계 |
|-----------|------|
| 응답 내용을 매번 LLM으로 검사 | 느리고 비용이 큼 |
| 각 에이전트 내부를 직접 모니터링 | 외부에서 가져온 블랙박스 에이전트에는 적용 불가 |
| 키워드/규칙 기반 필터 | 공격 문구가 조금만 바뀌어도 우회됨 |

**핵심 질문**
> 에이전트가 "무슨 말을 했는지" 보지 않고도, 공격당했는지 알 수 있을까?

# II. 제안 사항

---

## [슬라이드 6] 핵심 아이디어

**응답 "내용"이 아니라 "행동 패턴"을 본다**

공격을 받은 에이전트는 평소와 다르게 행동한다는 관찰에서 출발:
- 평소보다 응답이 느려지거나 빨라짐
- 평소보다 훨씬 길게/짧게 답함
- 이전 단계 대비 갑자기 튀는 변화가 생김

→ 이런 **통신 메타데이터**만으로도 이상 징후를 잡을 수 있지 않을까?

**실제로 정의한 6개 메타데이터 피처 (콘텐츠 비접근):**

| 피처 | 설명 |
|------|------|
| latency (δ) | 응답 지연 시간 |
| token_count (τ) | 응답 토큰 수 |
| api_freq (f) | 문장 수 등 호출 빈도 근사 |
| ctx_delta (Δc) | 이전 에이전트 대비 변화율 |
| call_seq (s) | 호출 이상 여부 플래그 |
| refusal_flag (r) | 거부 응답 감지 여부 |

**장점:** 내용을 안 보므로 빠르고, 개인정보 문제도 없고, 어떤 언어/어떤 LLM이든 동일하게 적용 가능

---

## [슬라이드 7] 그래프 구조 접근의 필요성

**에이전트 하나씩 따로 보면 애매하다**

```
Orchestrator: 살짝 이상한 것 같기도 하고...
Researcher:   음, 이 정도는 정상 범위 같은데...
```

**하지만 파이프라인 전체를 연결해서 보면 뚜렷해진다**

```
Orchestrator(이상) → Researcher(약간 이상) → Analyst(약간 이상) → Writer(정상)
   "앞에서 시작된 이상 신호가 점점 퍼져나가는 패턴"
```

→ 에이전트 각각을 독립적으로 보는 대신, **누가 누구에게 정보를 전달하는지(관계)** 를 그래프로 표현하고, 이 관계 구조 자체를 학습하자는 것이 이 연구의 방향

---

## [슬라이드 8] 경량 그래프 오토인코더 제안

**학습 방식 — 정답 라벨 없이 학습**
```
1단계: 정상적으로 돌아가는 파이프라인 패턴을 모델에게 보여줌
2단계: 새로운 세션이 들어오면, 그 패턴을 재현해보려 시도
       → 잘 재현되면 정상
       → 잘 재현이 안 되면 평소와 다른 상황 → 공격 의심
```

**설계 방향**
- 에이전트 = 그래프의 노드, 정보 전달 관계 = 그래프의 엣지
- 그래프 신경망(GNN) 기반의 아주 작은 오토인코더 구조 사용
- 목표: 무겁지 않고, 실시간으로 돌아가는 탐지기

**실제 설계한 모델 구조:**
```
Input (6개 피처)
  ├─ GCN Layer 1:  6 → 16
  ├─ GCN Layer 2: 16 →  8
  ├─ Decoder 1:    8 → 16
  └─ Decoder 2:   16 →  6   ← 원본 재구성

총 파라미터: 461개 (매우 가벼운 모델)
```

# III. 실험

---

## [슬라이드 9] 검증 방향

**두 단계로 아이디어를 검증하는 중**

1. **시뮬레이션 환경** — 다양한 공격 시나리오를 인위적으로 만들어, 이 방식이 공격 유형별로 잘 작동하는지 폭넓게 테스트
2. **실제 LLM 환경** — 실제로 로컬 LLM 여러 개를 파이프라인으로 연결해, 진짜 환경에서도 통하는지 확인

**공격 시나리오 5가지를 설계하여 검증 중**

| 유형 | 특징 |
|------|------|
| Direct | 즉시 역할을 바꾸려는 노골적 공격 |
| Harvest | 정보를 수집해 하위로 전파하는 공격 |
| Slow | 서서히 조금씩 오염시키는 공격 |
| Flood | 여러 에이전트를 동시에 오염시키는 공격 |
| **Chain** | **한 곳만 뚫어서 뒤로 전파시키는 공격 — 그래프 방식의 강점이 가장 크게 드러나는 유형** |

---

## [슬라이드 10] 초기 검증 결과

**탐지 성능 (AUC, 1에 가까울수록 좋음):**

| 실험 환경 | 탐지 성능 |
|-----------|:---------:|
| 시뮬레이션 (5-agent, 5가지 공격 유형) | 0.99 이상 |
| 실제 로컬 LLM (4-agent, llama3.2) | 최대 1.00 |

- 특히 **Chain 유형**(한 곳만 뚫려서 전파되는 공격)에서, 관계 구조(그래프)를 보는 방식이 그렇지 않은 방식보다 뚜렷하게 더 잘 잡아냄
- 실제 로컬 LLM으로 재현한 환경에서도, 공격의 파급력(전파 정도)이 충분히 클 때 안정적으로 탐지됨

**어느 에이전트가 침해됐는지도 어느 정도 짚어낼 수 있는 가능성 확인:**
```
Orchestrator  ████░░░░░░  낮음   ← 정상
Planner       ██████████  높음   ← ★ 침해 진원지로 추정
Researcher    ██████░░░░  중간   ← 전파 흔적
Writer        ████░░░░░░  낮음   ← 정상
```

> 아직 초기 검증 단계이며, 더 다양한 LLM·공격 방식으로 일반화 검증이 필요한 상태

# IV. 결론

---

## [슬라이드 11] 연구의 기대효과

**만약 이 아이디어가 잘 작동한다면:**

- 에이전트 내부나 응답 내용에 접근하지 않고도 배포 가능한 탐지 방식
- 외부에서 가져다 쓰는 블랙박스 MAS에도 적용 가능
- 매우 가벼운 모델이라 실시간 탐지에 적합
- 공격받은 지점을 짚어낼 수 있다면, 그 에이전트만 격리하고 나머지는 계속 운영하는 것도 가능
