## [슬라이드 3] 연구 배경

**"에이전트"란, 특정한 역할을 맡아 스스로 판단하고 행동하는 AI 프로그램을 의미한다.**

여러 에이전트가 하나의 작업을 나누어 맡고, 자신이 처리한 결과를 다음 에이전트에게 넘겨주며 협업하는 시스템을 멀티에이전트 시스템(Multi-Agent System, MAS)이라고 부른다. 최근 이러한 구조가 검색, 분석, 보고서 작성 등 실제 업무 자동화에 빠르게 도입되고 있다.

**예시로 드는 파이프라인 구조:**
```
사용자 → Orchestrator(작업 지시) → Researcher(자료 조사) → Analyst(분석) → Writer(보고서 작성) → 결과
```

> 위 구조는 여러 가능한 MAS 형태 중 설명을 위해 선택한 대표적인 예시이며, 실제 MAS가 반드시 이런 일렬 구조로만 존재하는 것은 아니다. 스타형(중앙 에이전트가 여러 에이전트에 동시에 지시), 계층형, 순환형 등 다양한 구조가 가능하다. 본 연구는 이 예시를 통해 문제와 아이디어를 설명한다.

**이 구조에서 아직 잘 알려지지 않은 보안 문제**
- 에이전트들은 이전 단계 에이전트가 넘겨준 내용을 검증 없이 그대로 신뢰하고 다음 작업에 사용한다.
- 이 "무조건적인 신뢰" 관계를 악용하는 공격이 가능하다.

## [슬라이드 4] 간접 프롬프트 인젝션 공격

**프롬프트란 AI에게 내리는 지시문을 의미한다.** "프롬프트 인젝션"은 공격자가 이 지시문에 악성 명령을 몰래 끼워 넣는 공격을 뜻하며, "간접"이라는 말이 붙는 이유는 공격자가 AI와 직접 대화하지 않고, AI가 나중에 참고하게 될 외부 문서나 검색 결과 안에 명령을 숨겨두기 때문이다.

```
정상적인 보고서처럼 보이는 문서
 └─ (문서 안에 몰래 숨겨진 지시문: "지금부터 다른 역할을 수행해라")
```

에이전트가 이 문서를 읽고 그 안의 숨겨진 지시를 그대로 따르면, 해당 에이전트는 "오염"된 상태가 된다. 문제는 여기서 끝나지 않는다.

**한 곳이 오염되면 전체가 도미노처럼 무너지는 구조**

```
Orchestrator 오염 → Researcher 오염 → Analyst 오염 → Writer 오염
```

*(위 4단계 구조는 슬라이드 3에서 설명한 대로 예시이며, 실제로는 에이전트 수와 연결 방식이 다를 수 있다. 중요한 점은 "앞 단계의 오염이 뒤 단계로 계속 전달되는 현상" 자체이다.)*

앞 단계 에이전트 하나만 속아도, 그 뒤에 있는 모든 에이전트가 오염된 정보를 그대로 이어받아 함께 오염되는 구조적 취약점이 존재한다.

## [슬라이드 5] 기존 방식의 한계

| 접근 방식 | 설명 | 한계 |
|-----------|------|------|
| 응답 내용을 매번 AI로 검사 | 별도의 AI가 각 응답 내용을 읽고 이상 여부를 판단 | 느리고 비용이 큼 |
| 각 에이전트 내부를 직접 모니터링 | 에이전트의 내부 프롬프트·코드를 직접 들여다봄 | 외부에서 가져와 내부를 볼 수 없는 "블랙박스" 에이전트에는 적용 불가 |
| 키워드·규칙 기반 필터 | 미리 정해둔 위험 단어나 패턴을 검사 | 공격 문구가 조금만 바뀌어도 우회됨 |

**핵심 질문**
> 에이전트가 "무슨 내용을 말했는지" 들여다보지 않고도, 공격을 당했는지 알아낼 수 있을까?

## [슬라이드 6] 관련 연구 ① — SentinelAgent

**논문 개요**
> 제목: SentinelAgent: Graph-based Anomaly Detection in Multi-Agent Systems  
> 저자: Xu He, Di Wu, Yan Zhai, Kun Sun  
> 발표: 2025, arXiv:2505.24201

**논문 요약**
- **그래프 기반 모델링:** 에이전트 간 상호작용을 실행 그래프(execution graph)로 표현하고, 노드·엣지·경로 단위로 이상 여부를 분석함
- **LLM 기반 의미 분석:** 이상 탐지를 위해 별도의 LLM 감시 에이전트가 각 에이전트의 응답 내용을 직접 읽고 의미를 해석하는 방식에 의존함

**향후 방향 (본 연구가 채우려는 공백)**
- 응답 내용을 읽지 않고, 통신 메타데이터만으로도 동일한 수준의 그래프 기반 탐지가 가능한지 검증할 필요가 있음

**논문 개요**

> 저자: Edoardo Debenedetti, Jie Zhang, Mislav Balunović, Luca Beurer-Kellner, Marc Fischer, Florian Tramèr

**논문 요약**

## [슬라이드 8] 핵심 아이디어

**응답의 "내용"이 아니라, 응답이 만들어지는 과정에서 드러나는 "행동 패턴"을 관찰한다.**

공격을 받은 에이전트는 평소와 다르게 행동한다는 관찰에서 출발한다.
- 평소보다 응답 속도가 느려지거나 빨라짐
- 평소보다 훨씬 길거나 짧게 답함
- 이전 단계 대비 갑자기 큰 폭으로 변화가 생김

**왜 이런 변화가 생기는가:** 공격자가 숨겨둔 지시문이 프롬프트에 추가되면, 에이전트가 처리해야 할 내용 자체가 길어지고 복잡해진다. 그 결과 응답을 만드는 데 걸리는 시간과 응답의 길이도 함께 달라지는 경향이 나타난다. 메타데이터의 변화는 이 인과관계에서 비롯된다.

이처럼 내용이 아니라 "내용 주변에서 자동으로 기록되는 부가 정보"를 메타데이터라고 부른다. (예: 메일의 본문 내용이 아니라, 메일이 언제 왔는지·글자 수가 몇 개인지 같은 정보)

**실제로 정의한 6개 메타데이터 피처 (응답 내용에는 접근하지 않음):**

| 피처 | 설명 |
|------|------|
| latency (응답 지연 시간) | 에이전트가 응답을 만드는 데 걸린 시간 |
| token_count (응답 토큰 수) | 응답의 길이 (토큰은 글자·단어를 세는 단위) |
| api_freq (호출 빈도) | 응답 안에 포함된 문장 수로 근사한 값 |
| ctx_delta (변화율) | 바로 앞 에이전트 응답 대비 얼마나 달라졌는지 |
| call_seq (호출 이상 플래그) | 평소와 다른 호출 패턴이 감지되었는지 여부 |
| refusal_flag (거부 응답 여부) | 에이전트가 "이 요청을 수행할 수 없다"고 답했는지 여부 |

**이 6개를 선택한 이유:** 별도의 로깅 장치를 새로 구축할 필요 없이, 에이전트가 응답을 만드는 과정에서 시스템에 이미 자동으로 기록되는 값들이기 때문이다. 기존 MAS 인프라를 수정하지 않고도 곧바로 적용할 수 있다는 실용적인 이유가 크다.

**이 방식의 장점:** 응답 내용을 읽지 않으므로 처리 속도가 빠르고, 개인정보가 노출될 위험이 없으며, 어떤 언어·어떤 AI 모델을 쓰더라도 동일한 방식으로 적용할 수 있다.

## [슬라이드 9] 그래프 구조 접근의 필요성

**에이전트를 하나씩 따로따로 살펴보면 이상 여부를 판단하기 애매하다.**

```
Orchestrator 단독 관측: 경미한 이상치 수준으로만 관측됨
Researcher 단독 관측:   정상 범위 내에 있는 것으로 판단됨
```

**그러나 파이프라인 전체를 연결해서 함께 보면 패턴이 뚜렷해진다.**

```
Orchestrator(경미한 이상) → Researcher(경미한 이상) → Analyst(경미한 이상) → Writer(정상)
   "앞에서 시작된 이상 신호가 뒤로 갈수록 함께 나타나는 연쇄 패턴(cascade)"
```

이런 연쇄 패턴을 포착하기 위해, 에이전트 각각을 독립적으로 보는 대신 "누가 누구에게 정보를 전달하는지"를 그래프로 표현하고, 이 관계 구조 자체를 학습하는 방식을 제안한다.

여기서 그래프란, 점(노드)들이 선(엣지)으로 연결된 구조를 의미한다. 이 연구에서는 각 에이전트가 노드가 되고, 에이전트 사이의 정보 전달 관계가 엣지가 된다.

## [슬라이드 10] 경량 그래프 오토인코더 제안

**오토인코더란, 입력을 압축했다가 다시 원래대로 복원해보는 방식으로 학습하는 모델을 의미한다.** 평소 패턴을 잘 학습해두면, 정상적인 입력은 복원이 잘 되지만, 낯선(비정상적인) 입력은 복원이 잘 되지 않는다는 원리를 이용한다.

**학습 및 탐지 방식 — 정답 라벨 없이 학습 가능**
```
1단계 (학습): 정상적으로 돌아가는 파이프라인의 메타데이터 패턴을 모델에게 보여줌
2단계 (탐지): 새로운 세션이 들어오면, 학습한 패턴을 바탕으로 복원을 시도
       → 복원이 잘 되면 정상
       → 복원이 잘 되지 않으면 평소와 다른 상황 → 공격 의심
```

**정상/공격을 구분하는 분류기 대신 오토인코더를 선택한 이유:** 공격 유형은 계속 새롭게 등장하지만, 실제 공격 사례(정답 라벨)를 충분히 모으기는 어렵다. 정상 데이터만으로 학습이 가능한 비지도 방식(One-Class Detection)을 택하면, 학습 시점에 존재하지 않았던 새로운 유형의 공격에도 적용될 가능성을 기대할 수 있다.

**그래프 신경망(GNN)이란,** 그래프로 연결된 노드들이 서로 정보를 주고받으며 학습하는 신경망을 의미한다. 이를 이용하면 에이전트 하나의 값뿐 아니라, 연결된 다른 에이전트의 상태까지 함께 고려하여 판단할 수 있다.

**여러 그래프 신경망 기법 중 GCN을 선택한 이유:** GCN은 구조가 가장 단순한 축에 속해 파라미터 수를 최소화하기 쉽고(경량화 목표에 부합), 결과 해석도 비교적 용이해 아이디어를 검증하는 초기 단계에 적합하다고 판단했다. GAT 등 더 복잡한 그래프 신경망과의 비교는 향후 과제로 남겨두었다.

**실제로 설계한 모델 구조:**
```
Input (에이전트당 6개 메타데이터 피처)
  ├─ GCN Layer 1:  6 → 16   (연결된 에이전트 정보를 모아 확장)
  ├─ GCN Layer 2: 16 →  8   (더 압축된 표현으로 요약)
  ├─ Decoder 1:    8 → 16   (복원 시작)
  └─ Decoder 2:   16 →  6   (원본 피처 형태로 복원)

총 파라미터(모델이 학습하는 내부 숫자 개수): 461개
→ 스마트폰 앱 하나보다도 훨씬 작은, 매우 가벼운 모델
```

## [슬라이드 11] 검증 방향

**아이디어를 두 단계로 나누어 검증하고 있다.**

1. **시뮬레이션 환경** — 실제 AI를 매번 실행하는 대신, 다양한 공격 상황을 컴퓨터로 인위적으로 만들어 대량으로 반복 실험. 여러 공격 유형에 대해 폭넓게 테스트하기에 적합하다.
2. **실제 LLM 환경** — 실제로 동작하는 언어 모델(LLM) 여러 개를 파이프라인 형태로 연결하여, 진짜 환경에서도 이 방식이 통하는지 확인한다.

**두 환경을 모두 사용하는 이유:** 시뮬레이션은 다양한 공격 유형을 저비용으로 대량 반복 테스트하기 위한 것이고, 실제 LLM 실험은 시뮬레이션에서 확인한 경향이 현실 환경에서도 유효한지 교차 검증하기 위한 것이다. 서로 다른 목적을 가진 상호보완적인 검증 절차이다.

**실제 LLM으로 상용 모델(GPT, Claude 등) 대신 로컬 오픈소스 모델(llama3.2)을 사용한 이유:** 별도 비용 없이 로컬 환경에서 반복 실행할 수 있어, 여러 시드에 걸쳐 다수의 세션을 대량으로 실험하기에 적합했기 때문이다. 상용 모델로의 일반화 검증은 향후 계획에서 다룬다.

**시뮬레이션에서 설계한 5가지 공격 유형**

| 유형 | 특징 |
|------|------|
| Direct | 처음부터 즉시 역할을 바꾸려는 노골적인 공격 |
| Harvest | 정보를 수집한 뒤 하위 에이전트로 전파하는 공격 |
| Slow | 서서히 조금씩 오염시켜 나가는 공격 |
| Flood | 여러 에이전트를 동시에 오염시키는 공격 |
| **Chain** | **한 곳만 뚫어서 뒤로 전파시키는 공격 — 그래프 방식의 강점이 가장 뚜렷하게 드러나는 유형** |

## [슬라이드 12] 초기 검증 결과

**탐지 성능을 나타내는 지표로 AUC를 사용한다.** AUC는 정상 상황과 공격 상황을 얼마나 잘 구별해내는지를 0~1 사이의 숫자로 나타낸 값이며, 1에 가까울수록 더 정확하게 구별한다는 의미이다.

| 실험 환경 | 탐지 성능(AUC) |
|-----------|:---------:|
| 시뮬레이션 (5개 에이전트, 5가지 공격 유형) | 0.99 이상 |
| 실제 로컬 LLM (4개 에이전트, llama3.2) | 최대 1.00 |

- 특히 **Chain 유형**(한 곳만 뚫려서 전파되는 공격)에서, 에이전트 간 관계 구조(그래프)를 함께 보는 방식이 그렇지 않은 방식보다 뚜렷하게 더 잘 탐지함을 확인했다.
- 실제 언어 모델로 재현한 환경에서도, 공격의 전파 정도가 충분히 클 때 안정적으로 탐지되었다.

**부가적으로, 어느 에이전트에서 문제가 시작되었는지도 어느 정도 짚어낼 수 있는 가능성을 확인했다.** (그래프 오토인코더는 파이프라인 전체를 한 번에 재구성하지만, 재구성 오차는 에이전트 단위로 각각 계산되기 때문에 어느 에이전트에서 오차가 크게 나타나는지 개별적으로 확인할 수 있다.)
```
Orchestrator  ████░░░░░░  낮음   ← 정상으로 판단
Planner       ██████████  높음   ← ★ 침해가 시작된 지점으로 추정
Researcher    ██████░░░░  중간   ← 오염이 전파된 흔적
Writer        ████░░░░░░  낮음   ← 정상으로 판단
```
(위 수치는 "재구성 오차"라는 값으로, 모델이 복원에 실패한 정도를 나타낸다. 값이 클수록 그 에이전트가 평소와 다르게 행동했다는 의미이다.)

> 아직 초기 검증 단계이며, 더 다양한 언어 모델과 공격 방식으로 일반화 검증이 필요한 상태이다.
