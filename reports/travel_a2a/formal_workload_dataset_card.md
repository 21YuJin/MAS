# Formal Workload Dataset Card — travel_a2a_v2

```
experiment_version: travel_a2a_v2
environment_type:   a2a_inspired_travel
graph_source:        interaction_events
llm_backend:         ollama
model_name:          llama3.2
workload_version:    travel_a2a_v2_formal_workload_v1
```

Covers `data/travel_a2a/formal_workload/` (50 task instances, 15 destinations,
15 destination-scoped content bundles) as of Step 6.5 (Phases A–D). This card
describes the workload as generated and behaviorally validated — it is
written from the actual generation/validation reports (`workload_summary.json`,
`phase_6_5d_validation_report.json`, etc.), not a plan.

## Dataset nature

- **Fully synthetic.** Every travel request, flight/hotel/tour option, and
  currency rate is generated by a deterministic, seeded procedural generator
  (`formal_workload_generator.py`, seed 42) — none of it is scraped, sampled,
  or derived from any real airline/hotel/tour operator or real booking system.
- **Not real prices or real availability.** Every content record carries
  `source_id: "generated_fixture"` and a `policy_note` stating this
  explicitly; `provider_id` values (e.g. `synthetic_hotel_3`) are placeholders,
  never real company names.
- **Not a claim about real travel-agent behavior.** The 5-agent
  (travel_coordinator/flight_agent/hotel_agent/currency_agent/tours_agent)
  A2A-inspired architecture and its deterministic `TravelWorkflowPolicy` are
  a research testbed, not a model of any deployed travel-booking product.

## Runtime nature

- **Structural generation is synthetic; execution telemetry is real
  where it runs on Ollama.** Phase 6.5D validated all 50 tasks via
  deterministic MOCK execution only (no Ollama calls) — this establishes that
  the workload's structural diversity produces genuinely diverse execution
  behavior (11 distinct event sequence patterns, 7 distinct graph structure
  patterns, 0 branch mismatches against each task's expected_normal_branches)
  before any real LLM cost is spent on it.
- Actual Ollama (llama3.2) execution against this workload — producing real
  token/timing telemetry, real orchestration events, and real Artifact
  lineage — is a **future, not-yet-performed** step (see
  `configs/travel_a2a/formal_workload/formal_collection_plan.json`, status
  `PLANNED_NOT_EXECUTED`).

## Purpose

- Feasibility study for **metadata-based indirect prompt injection
  detection** in a multi-agent LLM system (LightGAE), for a WISA 2026 poster.
- A **lightweight first-stage anomaly-triage** research testbed, not a
  production travel-recommender evaluation.
- Explicitly NOT a benchmark of travel-planning quality, itinerary
  correctness, or real-world booking accuracy.

## Known limitations (found and documented, not hidden)

- **Synthetic domain**: content is procedurally generated, not real-world
  travel data (see "Dataset nature" above).
- **One local model** (llama3.2 via Ollama) is the only LLM this workload has
  been (or will be) run against — no cross-model generalization claim.
- **Custom A2A-inspired orchestration**, not a standard/certified A2A
  implementation; no external booking tools are called (`content_repository.py`
  is a static, source-controlled fixture, not a live API).
- **Difficulty confounds, found during Phase 6.5D's mock full-run**:
  - `event_count` and `message_count` ranges are entirely non-overlapping
    across difficulty tiers (easy 6–8, medium 11–14, hard 16–19) — these raw
    counts alone would let a detector infer task difficulty without reading
    any injection-related signal.
  - 15/15 (100%) hard-difficulty tasks also exhibit the `budget_revision`
    branch — a structural consequence of `schedule_conflict` and
    `integration_conflict` being mutually exclusive in this content design
    (Step 6.5B), which forces every hard-tier task's 2-condition requirement
    to include `budget_conflict`. `hard_budget_confound_report.json` flags
    `recommend_redesign: true` by its own numeric rule.
  - **Decision**: Phase 6.5D identified these workload confounds. The formal
    workload remains usable for controlled feature engineering, but these
    confounds must be explicitly tracked during Step 7 and may require
    targeted workload redesign before final model evaluation. Concretely:
    Step 7's candidate-feature design may proceed against this workload, but
    `event_count`/`message_count`/`revision_count` must NOT be adopted as
    core detection features without controlling for difficulty (they
    currently separate difficulty tiers perfectly, independent of any
    injection signal), the hard-tier/`budget_revision` confound must be
    carried forward as its own tracked diagnostic rather than silently
    treated as resolved, and whether to redesign part of the hard-tier
    constraint mix should be revisited once Step 7's actual candidate
    features are known -- not decided permanently here.
  - `tours_agent` activation (29/50) sits just under the informal 30-session
    guideline; `currency_agent` (31/50) clears it.
  - Split-level branch-rate variance (>30 percentage points on some branches
    across train/validation/test) is plausible small-n sampling noise at
    validation/test sizes of 9–10, not a designed bias — flagged, not
    auto-corrected.
- **Task/content diversity is a controlled range**: 15 destinations, 4
  origins, 7 template families — not an open-world sample of all possible
  trips.

## Privacy

- No real user personal information of any kind appears anywhere in this
  workload — every `origin`/`destination`/traveler count/preference is a
  synthetic parameter drawn from a fixed catalog.
- Diagnostic/provenance fields (`expected_normal_branches`, `split`,
  `hard_normal_tags`, `generation_seed`, `generator_version`, `task_group_id`,
  `content_bundle_id`) are excluded from both the LLM-facing prompt input
  (`TaskInstance.to_travel_request_kwargs()`) and the LightGAE
  metadata-feature view (`FORBIDDEN_METADATA_KEYS`, verified by unit test in
  every one of Steps 2/5/6/6.5's test suites).
